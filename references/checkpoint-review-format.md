# 代码 checkpoint 分层复盘格式

仅在代码 checkpoint 已完成并有真实验证证据时使用。目标是让用户掌握“用了什么、如何工作、为什么这样设计”，并把项目实现转化为可迁移、能口述的工程知识。

不要用于每日计划、编码前讲解、普通问答、纯报错排查、尚未验证的半成品、纯文档或纯 skill 规则修改。

## L1：每个代码 checkpoint 必须提供

### 顶部

- `TL;DR`：一句话说明本次用了什么，以及为什么值得掌握。
- 概念标签：格式如 `dependency-injection·fallback·contract`。
- 回忆钩子：用一行复习上一个核心概念；用户答错或说“不会”的内容，在后续 checkpoint 主动复现。

### ① 关键问题

依次写出：

1. 编码前提出的核心问题；
2. 用户原回答；
3. 补全与纠正；
4. 可迁移的通用工程原理。

如果本 checkpoint 没有发生编码前问答，不要伪造用户回答；改为说明本次应掌握的核心判断及依据。

### ② 代码改动

每处改动都必须包含：

- 位置：真实文件路径，以及函数、类或紧凑行号；
- WHAT：使用了什么 API、模式、数据结构或契约；
- 代码：打印本次工作区中的真实代码片段；
- HOW：代码如何执行、数据如何流动、错误如何传播；
- WHY：为什么这样设计，并追溯到需求、设计或测试。

本节末尾写明：

- 本 checkpoint 实际实现的行为；
- 验证证据：真实命令或测试名，以及实际结果。

### ③ 技术面口述题

最多 3 道，每道包含：

- 要点答案；
- 一句话口述稿；
- 一个坏答样本，以及它为什么弱。

## L2：仅核心 1-2 处改动追加

- 通用原理；
- 生态映射：它与哪些已知技术或模式相似；
- 权衡备选：选择了什么，否掉了什么；
- 复杂度、性能或维护成本；
- 白板最小骨架；
- 换一个技术栈如何实现同一原理。

## L3：重要 checkpoint 或每天结束时追加

- 90 秒整合讲述；
- 5 分钟代码走读脚本；
- ASCII 数据流图；
- 风险到测试证据映射表。

## L4：跨 checkpoint 积累

- 累积概念标签，形成复习索引；
- 记录用户低信心或回答错误的概念，在后续主动复现；
- 技术概念使用中文解释，`reducer`、`Send`、API 名和标识符保持原名。

## 输出约束

- 代码片段、路径和验证结果必须来自当前工作区，不能从计划推测。
- 不单列与当前 checkpoint 无关的“未实现内容”或“下一 checkpoint”。
- 不展开无关权限和环境异常；只有它们实际阻塞验证时才简要说明。
- 默认使用简体中文；标识符、枚举值、error code 和 API 名保留英文。
