# 每日反馈与周计划自适应规则

生成下一天内容前，先根据用户反馈判断学习状态。不要机械推进周大纲。

新版自适应规则重点关注两类问题：

- 学习节奏问题：代码是否跑通、概念是否理解、任务是否超时。
- Agent 工程完整度问题：是否只是 demo，是否缺少 tool schema、权限、trace、eval、README、测试和失败记录。

## 每日模块配置自适应

每日教学默认使用 5 个模块，但模块结构本身也是计划的一部分，应接受用户配置：

- 用户未填写模块偏好时，直接使用默认 5 模块，不为此单独阻塞或追问。
- 在新周大纲和每日反馈模板中提供可选填写入口，引导用户说明保留、重命名、新增、删除和顺序。
- 用户配置默认持续作用于后续每日内容；如果用户只想改变下一天，记录为“仅本日”。
- 用户只调整学习难度或时间时，不要擅自改变模块结构。
- 用户自定义模块后，仍要检查核心目标、学习路径、代码产出、验收和次日反馈是否都有承载位置。
- 如果自定义结构漏掉必要能力，优先把能力嵌入现有模块；只有确实无法容纳时才提示用户补一个模块。

## 状态分类

### 正常推进

判断条件：

- 当天核心代码能运行。
- 用户能完成主要验收标准。
- 没有阻塞性 Bug。
- 本周产出物仍沿着项目阶梯推进。

处理方式：

- 明天按当前周大纲继续。
- 可以在每日内容中保留少量复习，但不改变主线。

### 放慢复习

判断条件：

- 核心代码没有跑通。
- 环境没有配好。
- 用户明确说概念没懂。
- 出现阻塞性报错，且用户无法定位。
- 用户完成了阅读，但没有写出可运行代码。

处理方式：

- 下一天降低 30%-50% 难度。
- 安排“复盘 + 最小修复 + 小步重写”。
- 不引入大型新框架或复杂新概念。
- 保留本周核心产出物，但缩小当天目标。
- 优先保证能运行、能打印中间状态、能解释数据流。

### 加难扩展

判断条件：

- 用户提前完成。
- 用户能讲清核心概念。
- 代码能运行，并完成扩展任务。
- 用户已经有测试或 README 初稿。

处理方式：

- 主线不跳跃。
- 给明天增加一个工程化扩展：
  - trace 日志。
  - eval case。
  - smoke test。
  - README。
  - `.env.example`。
  - 自定义异常类。
  - 权限确认。
  - 失败样例记录。
  - Git 分支或代码重构。

### 重排本周

判断条件：

- 连续两天卡住。
- Day 1 环境失败。
- 本周核心产出物已经明显无法按原计划完成。
- 用户学习时长连续低于计划的一半。
- 用户完成了太多阅读但没有形成可运行产出。

处理方式：

- 重排剩余 Day 内容。
- 缩小本周交付目标，但不取消可运行项目。
- 优先保证能跑、能讲、能 commit。
- 把“阅读大型项目源码”降级为“读 README + 画数据流 + 复刻最小模块”。

### Agent 项目完整度不足

判断条件：

- 只有一次 LLM API 调用，没有 loop。
- 只有 prompt，没有 tool schema。
- 只有工具函数，没有工具注册、参数校验和错误处理。
- 只有 RAG 问答，没有 citation、检索评估和失败处理。
- 只有多 agent 角色名，没有职责边界、输入输出 schema 和停止条件。
- 只有浏览器点击 demo，没有截图、DOM、动作日志和失败恢复。
- 没有 trace，无法解释每一步为什么发生。
- 没有 eval case，只凭一次 demo 判断效果。
- 没有 permission gate，对 shell、文件写入、网络访问等高风险操作没有边界。
- 没有 README、`.env.example`、测试、运行脚本或失败记录。

处理方式：

- 下一天优先补 agent 工程化，不继续堆新技术。
- 如果是 tool calling demo，补 tool registry、参数校验、结构化工具结果和错误反馈。
- 如果是 RAG demo，补 citation、检索日志、检索失败样例和 retrieval eval。
- 如果是 coding agent demo，补 permission gate、路径限制、命令白名单、trace 和 session store。
- 如果是 browser agent demo，补截图、DOM snapshot、action log、recovery rules 和安全限制。
- 如果不会讲项目，安排项目表达训练：问题场景、为什么需要 agent、数据流、工具边界、失败定位、eval 结果。

### 求职包装不足

判断条件：

- 项目能跑但简历说不清。
- README 没有业务背景、架构、运行方式、配置、示例输入输出和限制。
- 没有项目截图、trace 样例或 eval 结果。
- 用户无法解释与普通脚本、workflow、chatbot 的区别。

处理方式：

- 下一天安排项目包装，不继续堆框架。
- 输出简历 bullet、项目简介、技术亮点、难点和复盘问答。
- 如果目标是尽快投递，优先补 GitHub、README、演示链接、简历话术和个人优势。

## 内部判断清单

每日教学模式开始前，先内部判断：

- 当前周大纲是否存在；不存在则先生成或重建。
- 用户昨日反馈属于哪种状态。
- 今日安排是否仍服务于本周核心产出物。
- 今日是否需要补救、正常推进、扩展或重排。
- 当前项目是否至少有一个可运行入口。
- 当前项目是否有 tool schema 或清晰工具接口。
- 当前项目是否有状态记录或 trace。
- 当前项目是否有 eval/smoke test。
- 当前项目是否有权限边界。
- 当前项目是否有 README、`.env.example` 和失败记录。
- 今日是否需要补求职项目完整度：项目表达、技术亮点、投递材料。
- 当前每日模块配置是默认 5 模块、持续自定义，还是仅本日自定义。

如果需要调整，在 `📌 今日核心死磕` 开头用 1-2 句说明，例如：

“根据你昨天反馈 tool call JSON 能生成但工具执行没跑通，今天先不推进多工具 registry，改为把 calculator 的参数校验、错误反馈和 trace 彻底打牢。”

## 失败处理规则

用户贴出报错时：

- 先让报错回到最小复现，不直接重写整套项目。
- 解释可能根因，按概率排序。
- 给一个最小排查清单。
- 今日代码实战优先修复同类错误。
- 每日验收第 3 题必须改成 Bug 排查题。

同一问题连续两次仍未解决时：

- 不继续沿用同一修复思路。
- 回退到更小输入、更少依赖、更短调用链。
- 必要时把当天目标改成“生成最小可复现 + 定位失败层级”。
